# Agent Instructions for OpenCode Only

> **هذه التعليمات مخصصة لبرنامج OpenCode والنماذج (models) المربوطة به فقط.**
> **لا تنطبق على أي برامج أو نماذج أخرى.**
>
> فرع `اوبن-كود` وكل القواعد الواردة هنا خاصة ببرنامج OpenCode ووكلائه فقط.
> لا يحق لأي برنامج أو نموذج آخر (مثل Codex وغيره) استخدام فرع `اوبن-كود`
> أو اتباع هذه التعليمات؛ لكل برنامج فروعه وقواعده المستقلة.

---

## الهدف الأساسي

يجب أن تظل كل تعديلات OpenCode ظاهرة للمستخدم داخل **GitHub Desktop** قبل عمل
`commit` أو `push`. لا يجوز رفع التعديلات أو دمجها تلقائيًا.

## مكان العمل والفرع

- اعمل داخل النسخة الأساسية فقط:
  `/Users/mac/Documents/GitHub/FrameID`
- ممنوع إنشاء `worktree` أو clone مؤقت أو العمل من مسار آخر.
- ممنوع العمل مباشرة على `main`.
- استخدم الفرع المحلي `اوبن-كود` فقط:
  - إذا لم يكن موجودًا: `git switch -c اوبن-كود main`
  - إذا كان موجودًا: `git switch اوبن-كود`
- لا تحذف فرع `اوبن-كود` بعد الانتهاء؛ يجب أن يظل ظاهرًا في GitHub Desktop.
- إذا وجدت تعديلات غير مكوميتة، حافظ عليها ولا تعمل reset أو stash أو checkout
  لأي ملف إلا بطلب صريح من المستخدم.

## سير العمل الإلزامي

1. قبل التعديل، اعرض الفرع والحالة باستخدام `git status --short --branch`.
2. عدّل الملفات المطلوبة واختبرها.
3. بعد انتهاء التعديل، اعرض ملخصًا ونتيجة الاختبارات ثم **توقف للمراجعة**.
4. اترك الملفات غير مكوميتة لكي تظهر تحت **Changes** في GitHub Desktop.

## بوابات الموافقة

كل خطوة من الخطوات التالية تحتاج طلبًا صريحًا ومستقلًا من المستخدم:

- `commit`: لا تنفذه إلا عندما يقول المستخدم بوضوح «اعمل commit».
- الدمج مع `main`: لا تنفذه إلا عندما يقول المستخدم بوضوح «ادمج مع main».
- رفع `main`: لا تنفذه إلا عندما يطلب المستخدم رفع `main` صراحة
  (مثلًا «ارفع» أو «ادفع» أو «ارفع ال main»).

الموافقة على التعديل لا تُعتبر تلقائيًا موافقة على `commit` أو الدمج أو الرفع.

## سير الدمج والرفع (محليًا من GitHub Desktop)

الهدف: ألّا يحتاج المستخدم للدخول إلى GitHub على الويب لعمل `merge`. يُدمج العمل
محليًا ثم يُرفع من `main` مباشرةً:

1. عند طلب «ادمج مع main»:
   - تأكد أولًا أن كل تعديلات `اوبن-كود` مكوميتة وأن المستخدم راجعها في
     GitHub Desktop (تبويب **Changes/History**).
   - انتقل إلى `main`، حدّثه من `origin/main`:
     `git switch main && git pull --ff-only origin main`
   - ادمج `اوبن-كود` داخل `main`:
     `git merge --no-ff اوبن-كود -m "دمج فرع اوبن-كود"`
   - لا ترفع تلقائيًا بعد الدمج؛ توقف وانتظر طلب الرفع. اترك المستخدم يرى نتيجة
     الدمج في GitHub Desktop (تبويب **History** على `main`).
2. عند طلب «ارفع» (أو ما يعادله):
   - تأكد أنك على `main` وأنه محدّث.
   - ارفع فقط: `git push origin main`
   - لا ترفع `اوبن-كود` إلا إذا طلب المستخدم صراحةً رفع الفرع نفسه.

### قواعد إضافية
- لا تعمل merge وpush في خطوة تلقائية واحدة.
- لا تنفّذ الدمج إلا بعد تأكيد المستخدم أن التعديلات مكوميتة ورُوجعت.
- لا تدفع مباشرةً إلى `main` قبل انتهاء الدمج محليًا وتأكيد المستخدم للرفع.
- لا تحذف فرع `اوبن-كود` محليًا أو من GitHub بعد الدمج إلا بطلب صريح؛ يجب أن
  يظل ظاهرًا في GitHub Desktop.

## GitHub Desktop

- قبل الـcommit: تظهر الملفات في تبويب **Changes**.
- بعد الـcommit: تنتقل التعديلات إلى تبويب **History**؛ هذا سلوك طبيعي.
- بعد الـpush: يظهر الفرع والـcommits على GitHub.
- إذا لم تظهر التعديلات، تحقق أن GitHub Desktop مفتوح على نفس المسار الأساسي
  المذكور أعلاه وأن الفرع الحالي هو `اوبن-كود`.

## الأمان

- ممنوع وضع GitHub token أو كلمة مرور داخل رابط `origin`.
- يجب أن يكون رابط `origin` بالشكل التالي فقط:
  `https://github.com/badrabdoph1/FrameID.git`
- استخدم GitHub Desktop أو مدير بيانات الاعتماد للمصادقة.

## عناوين الـcommits

- عنوان الـcommit يطابق عنوان الطلب أو يصف التغيير بوضوح.
- لا تنشئ commit تجريبيًا فقط لاختبار الرفع.
