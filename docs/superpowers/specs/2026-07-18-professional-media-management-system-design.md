# تصميم نظام إدارة الوسائط وإطار العمليات طويلة التنفيذ

تاريخ التقرير: 2026-07-18

## الهدف

إنشاء قسم مستقل داخل لوحة الأدمن باسم "إدارة الوسائط" يكون المرجع الإداري الموحد لكل ملفات الوسائط داخل المنصة، مع الحفاظ على السلوك الحالي للتخزين والنسخ الاحتياطية في الإصدار الأول.

النظام ليس أداة حذف صور. هو طبقة إدارة وصيانة ومراقبة للوسائط، مبنية فوق:

- Storage Provider مستقل لا يعرفه منطق الأعمال.
- Operation Framework دائم لإدارة الأعمال طويلة التنفيذ.
- Media Catalog يوحد الرؤية بين قاعدة البيانات، uploads، GitHub الحالي، والنسخ الاحتياطية.
- Media Lifecycle واضح وآمن يمنع الحذف المباشر.

## المبادئ المعمارية

- لا يوجد أي منطق أعمال يعتمد على GitHub مباشرة.
- GitHub الحالي يبقى Provider ضمن واجهة تخزين موحدة.
- لا يتم تغيير طريقة النسخ الاحتياطية الحالية في هذه المرحلة.
- فحص الوسائط عملية قراءة وتحليل فقط، ولا تنفذ حذفًا أو نقلًا.
- أي تنظيف أو حذف نهائي يمر عبر خطة، تقرير، وتأكيد صريح.
- العمليات الطويلة لا تعتمد على جلسة الأدمن أو بقاء الصفحة مفتوحة.
- كل عملية قابلة للتتبع، الاستكمال، الإيقاف التعاوني، والسجل الكامل.

## الوضع الحالي المختصر

المشروع يحتوي على أكثر من مصدر للوسائط:

- `MediaAsset` لوسائط العملاء.
- `PlatformMediaAsset` لوسائط صفحات المنصة.
- ملفات ثابتة داخل `public/platform` و `public/uploads`.
- مراجع داخل JSON وحقول URL مثل القوالب، أقسام الموقع، إعدادات الثيم، محتوى الصفحات، إعدادات السوشيال، وحزم الخدمات.

المسار الحالي لرفع وسائط العملاء يستخدم `MediaStorageAdapter.save` فقط، وبعض المزودات تكتب محليًا ثم تحاول commit إلى GitHub. هذا مناسب كسلوك قائم، لكنه غير كاف كنظام إدارة وسائط لأن القراءة، الفحص، الجرد، الحذف، والتحقق ليست خلف واجهة موحدة حتى الآن.

يوجد أيضًا تنظيف مبدئي في `media-cleanup-service.ts`، لكنه:

- يفحص جزءًا محدودًا من المراجع.
- لا يدعم سلة محذوفات حقيقية.
- لا يحفظ خطة تنظيف أو سجل عمليات شامل.
- يتعامل مع HTTP URLs بشكل مختلف عن الملفات المحلية.
- لا يحسب Hash أو يكشف التكرارات بشكل موثوق.

## التعارض المهم

يوجد تعارض يجب توثيقه وعدم حله بانتقال فعلي الآن:

النظام الحالي يستطيع رفع بعض وسائط العملاء إلى GitHub main كملفات داخل `public/uploads` عند تفعيل إعدادات GitHub، بينما وثائق النسخ الاحتياطي تشير إلى أن بيانات العملاء يجب أن تبقى ضمن مسارات النسخ الاحتياطية، وأن `FULL backup` مسؤول عن `public/uploads`.

القرار في هذه المرحلة:

- لا ننقل التخزين إلى مزود جديد.
- لا نغير خطة النسخ الاحتياطي الحالية.
- لا نربط النظام الجديد بـ GitHub كمنطق أعمال.
- نضع GitHub خلف Storage Provider.
- نكشف التعارض داخل تقرير سلامة البيانات حتى يصبح مرئيًا وقابلًا للمعالجة لاحقًا.

## المعمارية المقترحة

النظام يتكون من أربع طبقات رئيسية:

1. Storage Layer

واجهة موحدة لكل عمليات التخزين. المزود الحالي GitHub يصبح تنفيذًا خلف هذه الواجهة، ويمكن لاحقًا إضافة Railway Volume أو S3 أو Cloudinary بدون تعديل منطق الوسائط.

العقد المقترح:

- `putObject`
- `getObjectMetadata`
- `readObject`
- `listObjects`
- `deleteObject`
- `resolvePublicUrl`
- `capabilities`

لا تحتوي الواجهة على قرارات مثل "هل الملف مستخدم؟" أو "هل يجوز حذفه؟". هذه قرارات Media Domain.

2. Operation Framework

إطار عام لكل الأعمال الطويلة:

- فحص الوسائط.
- تنظيف الوسائط.
- إعادة المزامنة.
- إصلاح البيانات.
- إنشاء thumbnails مستقبلًا.
- ضغط الصور.
- التحقق من سلامة النسخ الاحتياطية.

كل عملية تسجل في قاعدة البيانات وتعمل على دفعات، وتحفظ تقدمها دوريًا. عند إعادة تشغيل التطبيق أو حدوث Deploy، تنتهي صلاحية lease العملية ويمكن لعامل جديد استكمالها من آخر checkpoint.

3. Media Catalog

فهرس إداري موحد يربط الوسائط بمكانها ومصدرها ومراجع استخدامها. لا يلغي الجداول الحالية فورًا، بل يعمل كطبقة سيطرة تدريجية فوقها.

الفهرس يجب أن يعرف:

- أصل العنصر: customer media, platform media, static file, discovered object.
- Provider التخزين و `storageKey`.
- الحجم، النوع، الأبعاد، Hash، تاريخ الاكتشاف، آخر تحقق، آخر استخدام.
- حالة دورة الحياة وحالة الاستخدام.
- مراجع الاستخدام التفصيلية.

4. Admin UI

قسم "إدارة الوسائط" يحتوي على:

- لوحة إحصائيات.
- فحص الوسائط.
- نتائج الفحص.
- سلة المحذوفات.
- التكرارات.
- سلامة البيانات.
- سجل العمليات.

## Operation Framework

الحالات المقترحة للعملية:

- `PENDING`
- `RUNNING`
- `PAUSE_REQUESTED`
- `PAUSED`
- `CANCEL_REQUESTED`
- `CANCELLED`
- `SUCCEEDED`
- `PARTIAL`
- `FAILED`

الحقول الأساسية:

- معرف العملية.
- نوع العملية.
- الحالة.
- نسبة الإنجاز.
- إجمالي العناصر.
- عدد العناصر المنتهية.
- وقت البدء والانتهاء.
- آخر تحديث.
- الأدمن الذي بدأ العملية.
- إمكانية الإيقاف.
- إمكانية الاستكمال.
- checkpoint.
- النتيجة النهائية.
- سجل الأخطاء.
- lease owner و lease expiry.

آلية التشغيل المقترحة للإصدار الأول:

- Queue مبني على قاعدة البيانات.
- Runner داخلي داخل التطبيق مع lease و heartbeat.
- Endpoint أو tick داخلي لإيقاظ العمليات عند الحاجة.
- قابل للانتقال لاحقًا إلى Railway Worker منفصل دون تغيير API العمليات.

البدائل:

- Runner داخلي فقط: أبسط وأقل تكلفة، لكنه يشارك موارد تطبيق الويب.
- Railway Worker منفصل: أفضل للعزل والتوسع، لكنه يضيف خدمة تشغيل جديدة.
- Queue خارجي مثل Redis/BullMQ: قوي، لكنه تعقيد غير مبرر الآن.

التوصية: DB-backed Operation Framework مع runner داخلي في البداية، وتصميم يسمح بفصله لاحقًا.

## Media Lifecycle

لا يتم تمثيل الوسائط فقط بـ "محذوف/غير محذوف". يوجد فصل بين دورة الحياة وحالة الاستخدام.

حالات دورة الحياة:

- `ACTIVE`
- `IN_TRASH`
- `PURGED`

حالات الاستخدام:

- `USED`
- `UNUSED`
- `UNKNOWN`

`ELIGIBLE_FOR_PURGE` ليست حالة مخزنة أساسية، بل حالة مشتقة عندما يكون العنصر:

- داخل السلة.
- انتهت مدة الاحتفاظ به.
- غير مستخدم حسب آخر فحص.
- لا تظهر له مراجع حالية عند إعادة التحقق قبل الحذف.

هذا الفصل يمنع مشكلة شائعة: "غير مستخدم" ليست حقيقة دائمة، بل نتيجة فحص قابلة للتغير.

## سياسة سلة المحذوفات

القيمة الافتراضية لمدة الاحتفاظ: 30 يومًا.

القيمة قابلة للتعديل من إعدادات النظام. عند نقل عنصر إلى السلة يتم حفظ `purgeEligibleAt` طبقًا للإعداد الحالي وقت النقل، حتى تكون المواعيد قابلة للتفسير. تغيير الإعداد لاحقًا يؤثر على العناصر الجديدة فقط، مع إمكانية إضافة إجراء إداري مستقبلي لإعادة حساب مواعيد العناصر الموجودة.

لا يوجد حذف نهائي تلقائي في الإصدار الأول.

صفحة السلة تعرض:

- البحث والفلترة.
- المعاينة.
- تاريخ النقل للسلة.
- السبب.
- المستخدم أو العملية.
- الأيام المتبقية قبل الأهلية.
- الحجم المتوقع استرداده.
- أماكن الاستخدام السابقة عند توفرها.
- الاستعادة الفردية والجماعية.
- الحذف النهائي الفردي والجماعي بعد تأكيد صريح.

إذا عاد استخدام عنصر موجود في السلة أو مؤهل للحذف:

- يفقد أهلية الحذف فورًا.
- يعاد اعتباره `ACTIVE` و `USED`.
- يسجل الحدث في سجل دورة الحياة.

قبل الحذف النهائي:

- يتم فحص المراجع من جديد.
- إذا وجدت أي مرجع، يتم إلغاء حذف هذا العنصر فقط.
- تسجل النتيجة داخل Operation Log.
- العناصر الآمنة الأخرى يمكن أن تستمر كـ partial operation.

## فحص الوسائط

فحص الوسائط عملية Operation من نوع `MEDIA_SCAN`.

مصادر الفحص:

- سجلات `MediaAsset`.
- سجلات `PlatformMediaAsset`.
- مجلدات `public/uploads` و `public/platform`.
- Inventory التخزين من Storage Provider عند توفره.
- manifests النسخ الاحتياطية.
- مراجع قاعدة البيانات المباشرة.
- مراجع JSON داخل المحتوى والقوالب والثيمات.

نتائج الفحص:

- مستخدم.
- غير مستخدم.
- مكرر.
- تالف.
- مفقود في التخزين.
- موجود في التخزين بلا سجل.
- سجل يشير إلى ملف غير موجود.
- حجم كبير جدًا.
- أبعاد غير صحيحة.
- اختلاف بين DB و uploads و GitHub و backups.

الفحص لا يحذف ولا ينقل. هو يحدّث الفهرس، يحفظ findings، ويولد تقريرًا.

## التنظيف الذكي

التنظيف الذكي عملية من مرحلتين:

1. إنشاء خطة تنظيف من آخر Scan موثوق.
2. تنفيذ الخطة بعد تأكيد الأدمن.

الخطة تحفظ snapshot للمرشحين، سبب الترشيح، الحجم المتوقع توفيره، وآخر حالة استخدام معروفة.

قبل نقل أي عنصر إلى السلة:

- يعاد فحص الاستخدام الحالي.
- إذا ظهر استخدام، يتم تخطي العنصر وتسجيل السبب.
- إذا لم يظهر استخدام، ينتقل إلى `IN_TRASH`.

لا يوجد حذف نهائي ضمن عملية التنظيف. الحذف النهائي عملية مستقلة من سلة المحذوفات.

## إدارة التكرارات

يحسب النظام SHA-256 لكل عنصر عند الرفع أو عند الفحص.

إذا ظهر أكثر من عنصر بنفس Hash:

- لا يحذف النظام أي شيء.
- يعرض المجموعة، أماكن الاستخدام، الأحجام، والمكسب المتوقع.
- يقدم اقتراح دمج فقط عندما تكون المراجع قابلة للتحويل بأمان.
- قرار الدمج للأدمن، والتنفيذ Operation مستقل.

في الإصدار الأول يمكن الاكتفاء بالتقرير دون تنفيذ الدمج الفعلي.

## سلامة البيانات

تقرير السلامة يقارن:

- قاعدة البيانات.
- uploads المحلي.
- Provider الحالي.
- النسخ الاحتياطية.

الفروقات تظهر بوضوح:

- DB reference بدون object.
- object بدون DB record.
- object موجود في GitHub وغير موجود محليًا.
- object موجود محليًا وغير موجود في backup manifest.
- checksum mismatch.
- اختلاف حجم أو نوع.

هذا التقرير هو المكان الصحيح لإظهار تعارض GitHub الحالي مع سياسة النسخ الاحتياطية دون تغيير التخزين الآن.

## نموذج البيانات المقترح

جداول جديدة أو مكافئة لها:

- `Operation`
- `OperationEvent`
- `OperationItem`
- `MediaCatalogEntry`
- `MediaReference`
- `MediaFinding`
- `MediaLifecycleEvent`
- `MediaSettings`

لا يجب حذف أو دمج `MediaAsset` و `PlatformMediaAsset` في المرحلة الأولى. الفهرس الجديد يعمل بجوارهما ويقلل المخاطر.

## الأمان

- صلاحيات منفصلة للفحص، التنظيف، الاستعادة، والحذف النهائي.
- لا حذف جماعي دون خطة وتأكيد صريح.
- كل عملية destructive تحتاج operation id و confirmation token.
- كل purge يعيد فحص المراجع قبل حذف object من Provider.
- استخدام optimistic version أو snapshot revision لمنع تنفيذ خطة قديمة على حالة جديدة.
- الحذف النهائي idempotent: إذا حذف object ثم انقطع التنفيذ، تستكمل العملية وتغلق الحالة بأمان عند إعادة المحاولة.

## تصميم واجهة الأدمن

قسم "إدارة الوسائط" يكون عمليًا ومناسبًا للاستخدام المتكرر:

- إحصائيات علوية كثيفة وواضحة.
- Tabs أو sidebar فرعي: Overview, Scan, Trash, Duplicates, Integrity, Operations, Settings.
- جداول قابلة للبحث والفلترة.
- معاينة جانبية للوسيط.
- Progress واضح للعمليات الجارية.
- سجل مختصر داخل الصفحة وسجل كامل في تفاصيل العملية.
- Responsive للهاتف وسطح المكتب.

## مراحل التنفيذ

المرحلة 1: الأساس الآمن

- تقرير التصميم.
- أنواع Lifecycle و Usage.
- دوال أهلية الحذف والاستعادة والحماية قبل purge.
- عقود Operation Framework.
- عقود Storage Provider الجديدة.
- اختبارات وحدة للقرارات الحساسة.

المرحلة 2: قاعدة البيانات والفهرس

- إضافة جداول العمليات والفهرس.
- Repository للعمليات والفهرس.
- backfill scan مبدئي.

المرحلة 3: فحص الوسائط

- MEDIA_SCAN operation.
- Reference collectors.
- Storage inventory adapters.
- تقرير نتائج الفحص.

المرحلة 4: لوحة الأدمن

- صفحة إدارة الوسائط الجديدة.
- صفحة العمليات.
- صفحة سلة الوسائط.
- صفحة التكرارات والسلامة.

المرحلة 5: التنظيف والحذف النهائي

- خطط تنظيف.
- نقل للسلة.
- استعادة.
- purge بتأكيد صريح وإعادة فحص.

المرحلة 6: تحسينات مستقبلية

- thumbnails.
- medium versions.
- compression.
- duplicate merge.
- storage migration assistant.

## قرار الإصدار الأول

الإصدار الأول يجب أن يبدأ بالأساس الآمن القابل للاختبار، ولا يغير طريقة التخزين أو النسخ الاحتياطي الحالية. أي تشغيل فعلي للحذف النهائي يجب أن يأتي بعد توفر Media Catalog و Reference Registry و Operation Logs، لأن الحذف قبل ذلك سيكون قرارًا غير قابل للمراجعة الكافية.
