# Template Starter Content Design

## الهدف

كل موقع جديد يبدأ كنسخة مستقلة ومكتملة من القالب الذي اختاره العميل. تتطابق النسخة مع المعاينة في النصوص والصور والباقات والإضافات والتواصل وإعدادات الثيم، مع استبدال اسم المصور التجريبي فقط باسم صاحب الحساب.

## مصدر البيانات

يملك كل قالب `starterContent` معرفًا ومتحققًا منه داخل سجل القوالب. يستخدم المصدر نفسه في ثلاثة مسارات:

1. بناء معاينة القالب.
2. تجهيز بيانات إنشاء الموقع وقت التسجيل.
3. ملء حقول لوحة تحكم العميل بالقيم الفعلية المحفوظة.

لا تحتفظ صفحة المعاينة أو خدمة التسجيل بنسخة ثابتة أخرى من البيانات. يمكن لبيانات `Template.previewData` التي يعدلها الأدمن أن تحل محل بيانات البداية المسجلة بعد التحقق من اكتمالها، حتى تصبح المعاينة والموقع الجديد متزامنين مع آخر نسخة منشورة.

## نموذج بيانات البداية

يشمل النموذج:

- عنوان الموقع ووصفه وبيانات SEO.
- بيانات Hero والعناوين والنصوص وحالة ظهور الأقسام.
- بيانات التواصل وروابط السوشيال ورسالة الحجز.
- الباقات بكل خصائصها وترتيبها وصورها.
- الإضافات بكل خصائصها وترتيبها.
- ألبوم صور مبدئي وصور المعرض وترتيبها ووصفها.
- إعدادات الثيم.

الصور البعيدة المستخدمة في القالب تحفظ للموقع الجديد كسجلات `MediaAsset` مستقلة تخص Tenant العميل، ثم ترتبط بالألبوم وصورة الغلاف. لا يتم تنزيل الصورة أثناء التسجيل؛ يحتفظ السجل بعنوان الصورة المنشور مع بيانات وصفية واضحة، فيبقى التسجيل سريعًا ولا يعتمد على نجاح شبكة خارجية لحظيًا.

## تخصيص اسم العميل

تطبق خطوة تخصيص صريحة على حقول هوية المصور فقط: اسم الموقع، اسم الاستوديو، عنوان Hero، عنوان SEO، والاسم داخل structured data. لا يستخدم استبدال نصي شامل قد يغير نصوصًا غير مقصودة. تبقى بقية بيانات القالب مطابقة للمعاينة.

## تدفق التسجيل

1. التحقق من القالب المختار وأنه منشور.
2. تحميل بيانات البداية المنشورة والتحقق من بنيتها.
3. إنشاء نسخة مخصصة باسم صاحب الحساب في الذاكرة.
4. إنشاء المستخدم وTenant والموقع وكل المحتوى داخل معاملة قاعدة بيانات واحدة.
5. عند أي فشل تتراجع المعاملة كاملة ولا يبقى حساب أو موقع ناقص.

إذا لم يحدد المستخدم قالبًا يستخدم `noir-gold` بشرط أن يكون منشورًا وصالحًا. إذا كانت بيانات قاعدة البيانات قديمة أو ناقصة، تستخدم بيانات البداية المسجلة لنفس القالب. إذا لم يوجد أي مصدر مكتمل، يتوقف التسجيل برسالة واضحة بدل إنشاء موقع ناقص.

## الاستقلال والتوافق

ينسخ كل موقع سجلاته وقيم JSON الخاصة به. تعديل العميل لا يغير القالب أو مواقع العملاء الآخرين. المواقع القديمة لا ترحل تلقائيًا ولا تتغير. تبقى عقود عرض الموقع ولوحة التحكم الحالية كما هي، لكنها تستقبل بيانات مكتملة منذ البداية.

## حالات الفشل

- قالب غير موجود أو غير منشور: رفض التسجيل وطلب اختيار قالب متاح.
- بيانات قالب غير مكتملة أو غير صالحة: رفض آمن مع تسجيل سبب الخطأ.
- صورة بعيدة غير متاحة لاحقًا: تعرض واجهة الموقع fallback الحالي وتبقى بقية البيانات قابلة للتعديل.
- بريد أو هاتف مكرر: التوقف قبل بدء الإنشاء.
- رابط موقع متعارض: توليد رابط بديل وفق السياسة الحالية.
- فشل أي عملية كتابة: rollback كامل للمعاملة.

## معايير القبول

- معاينة كل قالب والموقع الجديد المبني منه ينتجان البيانات نفسها، باستثناء اسم المصور.
- حقول لوحة التحكم تعرض القيم المحفوظة نفسها، لا placeholders فارغة.
- توجد صور معرض فعلية وباقات وإضافات وبيانات تواصل بعد التسجيل.
- تعديل موقع عميل لا يؤثر على القالب أو عميل آخر.
- اختبارات الخدمة والمستودع وبذور المنصة والمعاينة تمر بنجاح.
- `typecheck` و`lint` و`test` و`build` تمر قبل دمج المرحلة.
