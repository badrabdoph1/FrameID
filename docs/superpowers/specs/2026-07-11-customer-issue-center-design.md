# تصميم نظام الأخطاء ومركز مشاكل العملاء

**التاريخ:** 2026-07-11  
**الحالة:** معتمد للتخطيط  
**الفرع:** `codex/نظام-الابلاغ`

## 1. الهدف

إعادة بناء تجربة الأخطاء في FrameID بحيث تبقى مطمئنة للعميل، بينما يحصل فريق الإدارة على بلاغ تقني كامل وقابل للمتابعة من لحظة ظهوره حتى الحل والإغلاق. يجب أن تعتمد كل أجزاء المنصة على مسار واحد للأخطاء والبلاغات، وأن تطوّر `ErrorLog` وError Center الموجودين بدل إنشاء نظام موازٍ.

## 2. الوضع الحالي

المشروع يملك بالفعل أساسًا صالحًا للتوسعة:

- `src/lib/errors` يحتوي على تصنيف الأخطاء، `processError()`، Logger، Request Context، وServer Action Wrapper.
- `ErrorLog` يسجل الأخطاء، ويحتوي على Request ID وCorrelation ID وبعض بيانات السياق.
- `/admin/errors` يعرض Error Center بسيطًا ويسمح بتعليم السجل كمحلول.
- Middleware يولد Request ID وCorrelation ID ويمرر المسار والطريقة والرابط.
- `/api/client-diagnostics` يجمع معلومات أجهزة وتقارير Rendering داخل `ErrorLog`.
- يوجد `SupportCase` بسيط، لكنه لا يمثل التفاصيل التقنية أو دورة العمل المطلوبة.
- صفحات `error.tsx` و`global-error.tsx` و`not-found.tsx` الحالية غير موحدة وتعرض أحيانًا Digest أو صياغات تقنية ومقلقة.

المشكلة الأساسية ليست غياب التسجيل، بل تشتت الالتقاط، وعدم اكتمال البيانات المحفوظة، وعدم وجود كيان بلاغ ودورة حياة واضحة، واختلاف تجربة العميل من مساحة إلى أخرى.

## 3. القرار المعماري

سيصبح النظام مكوّنًا من مستويين مرتبطين:

1. **Error Occurrence:** كل ظهور فعلي للخطأ يُلتقط تلقائيًا في `ErrorLog` بعد تنقية البيانات الحساسة. هذا سجل تقني وليس بالضرورة بلاغًا يحتاج تدخلًا بشريًا.
2. **Customer Issue:** بلاغ قابل للإدارة يُنشأ عندما يضغط المستخدم «إبلاغ الإدارة بالمشكلة»، أو عندما تصعّد سياسة السيرفر خطأ حرجًا إلى بلاغ داخلي. يرتبط البلاغ بواحد أو أكثر من سجلات `ErrorLog`.

بهذا القرار نحافظ على رؤية كل الأخطاء بدون إغراق مركز مشاكل العملاء. ويُمنع تكرار البلاغات عبر Fingerprint ثابت ونافذة دمج للبلاغات المفتوحة الخاصة بنفس العميل/الموقع.

لن يُستخدم `SupportCase` كبديل للبلاغ التقني؛ فهو يظل مخصصًا لمحادثات الدعم العامة. ويمكن ربطه مستقبلًا ببلاغ تقني عند الحاجة، لكن ذلك خارج نطاق هذه المرحلة.

## 4. نموذج البيانات

### 4.1 توسيع `ErrorLog`

يظل `ErrorLog` مصدر الحقيقة لكل ظهور تقني، ويُضاف إليه ما يلي:

- `issueId`: البلاغ المرتبط إن وجد.
- `fingerprint`: بصمة مبنية من نوع الخطأ والمسار وأول إطار صالح من Stack وDigest.
- `errorType`: اسم الخطأ أو نوع المصدر.
- `method`, `url`, `stack`, `digest`, `cause`.
- `tenantId`, `siteId`, `sessionId`, `adminUserId`.
- `environment`, `releaseVersion`, `buildVersion`.
- `sourceArea`: `MARKETING`, `CUSTOMER_DASHBOARD`, `ADMIN`, `PUBLIC_SITE`, `API`, `SERVER_ACTION`, أو `GLOBAL`.
- `sourceFile`, `sourceLine`, `sourceColumn` عند إمكان استخراجها بأمان.
- `ipAddress`, `userAgent`, `browser`, `device`, `os`, `language`, `timezone`, `screenSize`, `referrer`, `connectionStatus`.
- `lastAction` و`templateCode`.
- `metadata` للتفاصيل الإضافية المنقاة فقط.

تظل الحقول التقنية غير معروضة للمستخدم النهائي.

### 4.2 كيان `CustomerIssue`

يحتوي البلاغ على:

- رقم داخلي `id` ورقم عرض متسلسل `number` بصيغة مثل `ISS-000123`.
- `status`: `NEW`, `IN_REVIEW`, `RESOLVED`, `CLOSED`.
- `priority`: `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`.
- `source`: `CUSTOMER_REPORT`, `INTERNAL_AUTO`, `ADMIN_REPORT`.
- `title`, `fingerprint`, `customerNote` الاختيارية.
- `userId`, `tenantId`, `siteId`, `sessionId` عند توفرها.
- `assigneeAdminUserId`.
- `createdAt`, `updatedAt`, `reviewStartedAt`, `resolvedAt`, `closedAt`.
- `resolvedByAdminUserId`, `resolutionNote`.
- علاقة بسجلات `ErrorLog` التي تفسر البلاغ وتكراراته.

### 4.3 سجل حركة البلاغ

يُضاف `CustomerIssueEvent` لتسجيل كل تغيير مهم:

- إنشاء البلاغ أو دمج ظهور جديد.
- بدء المراجعة وتعيين المسؤول.
- تغيير الأولوية أو الحالة.
- الحل، إعادة الفتح، والإغلاق.
- إرسال إشعار للعميل.
- هوية الأدمن المنفذ ووقت الإجراء وبيانات مختصرة غير حساسة.

### 4.4 الإشعارات

يُوسّع مسار الإشعارات الحالي بدل إنشاء قناة منفصلة:

- إشعار أدمن غير مقروء عند إنشاء بلاغ جديد.
- Badge مبني على عدد بلاغات `NEW` وليس على عداد محلي قابل للاختلاف.
- عند الحل، يمكن إنشاء `Notification` للـ Tenant مع النص الافتراضي: «تم حل المشكلة التي أبلغت عنها، ويمكنك المحاولة مرة أخرى.»
- يسجل `CustomerIssueEvent` اسم الأدمن ووقت إرسال إشعار الحل.

## 5. خدمة الالتقاط الموحدة

تُقدّم خدمة واحدة بواجهات واضحة:

- `captureErrorOccurrence(input)`: تنقية وتصنيف وتسجيل ظهور الخطأ.
- `reportCustomerIssue(input)`: إنشاء بلاغ أو دمجه مع بلاغ مفتوح مطابق، وربط سجل الخطأ به.
- `transitionCustomerIssue(input)`: تنفيذ انتقال حالة مسموح وتسجيل Event.
- `notifyCustomerIssueResolved(input)`: إرسال إشعار الحل وتسجيله.

تستقبل الخدمة سياقًا موحدًا، لكن القيم الموثوقة مثل `userId`, `tenantId`, `siteId`, `sessionId`, IP، وبيئة التشغيل تُشتق على السيرفر. لا يُسمح للمتصفح بفرض هذه القيم داخل البلاغ.

## 6. مسارات الالتقاط

### 6.1 Error Boundaries وGlobal Error

- تستخدم كل ملفات `error.tsx` مكوّن عرض واحدًا.
- عند Mount، يُرسل ظهور الخطأ تلقائيًا إلى Endpoint الالتقاط مرة واحدة.
- يحتفظ المكوّن بمعرف الظهور في الذاكرة/Session Storage.
- زر الإبلاغ يرقّي الظهور إلى `CustomerIssue` بدون مطالبة المستخدم بكتابة شيء.
- `global-error.tsx` يستخدم نسخة مستقلة قليلة الاعتماد تعمل حتى عند تعطل Root Layout، لكنها تتبع نفس النص والألوان والأفعال وترسل إلى نفس Endpoint.

### 6.2 أخطاء المتصفح

Provider مركزي يسجل:

- `window.error`.
- `unhandledrejection`.
- أخطاء React Boundaries.
- آخر Action مفيد من قائمة محدودة: تنقل، إرسال نموذج، حفظ، رفع، دفع، نشر، أو ضغط زر موسوم بـ`data-error-action`.

يُحفظ آخر عدد محدود من Actions في Session Storage دون قيم حقول النماذج.

### 6.3 API وServer Actions

- يُوسّع `processError()` وServer Action Wrapper لاستخدام خدمة الالتقاط نفسها.
- توفر Route Handlers Wrapper يعيد استجابة عامة وآمنة مع Request ID داخلي، ويسجل التفاصيل التقنية على السيرفر.
- لا تُنشأ `CustomerIssue` تلقائيًا لكل 4xx متوقع أو Validation Error.
- أخطاء `FATAL` أو حالات محددة مثل تعطل قاعدة البيانات المتكرر يمكن تصعيدها إلى `INTERNAL_AUTO` مع Fingerprint وتهدئة زمنية تمنع التكرار.

### 6.4 المواقع المنشورة

- يُستخرج `siteId` و`tenantId` من slug/host وسياق الموقع الموثوق.
- تُسجل Template Code وTemplate Version.
- لا يشترط تسجيل الدخول للإبلاغ من موقع منشور؛ تُربط البيانات بالموقع، وتظل هوية العميل فارغة إن لم توجد جلسة.
- يخضع Endpoint لمحدد معدل وDeduplication لمنع إساءة الاستخدام.

## 7. Fingerprint ومنع التكرار

تُحسب البصمة على السيرفر من قيم مستقرة بعد إزالة الأرقام والمعرفات المتغيرة من الرسالة:

- نوع الخطأ المصنف.
- Source Area.
- Route normalized.
- أول إطار Stack داخل المشروع، أو Digest عند غياب Stack.
- Error Code الداخلي.

عند ضغط الإبلاغ، يبحث النظام عن بلاغ غير مغلق يحمل البصمة نفسها ونفس `tenantId/siteId/userId` خلال 24 ساعة. إذا وُجد، يربط الظهور الجديد به ويحدث `updatedAt` بدل إنشاء بلاغ آخر. البلاغات الداخلية الحرجة تستخدم نافذة تهدئة قابلة للضبط منفصلة.

## 8. تنقية البيانات والأمان

قبل التخزين، تمر كل Metadata عبر Sanitizer مركزي:

- حذف كلمات المرور، Tokens، Cookies، Authorization Headers، أسرار API، CSRF، رموز إعادة التعيين، ومحتويات حقول الدفع.
- إزالة Query Parameters الحساسة من URL وReferrer.
- تحديد أطوال الرسالة وStack وMetadata وحجم Payload.
- قبول IP من Headers البنية التحتية الموثوقة فقط، مع عدم عرضه خارج الأدمن.
- عدم جمع محتوى حقول النماذج أو نصوص المستخدم كجزء من آخر Action.
- Rate limiting وOrigin validation وCSRF protection حيث تنطبق.
- صلاحيات الأدمن تحدد من يرى Stack وIP وMetadata ومن يغيّر الحالة.

## 9. تجربة صفحات الأخطاء

### 9.1 اللغة

الرسالة العامة المعتمدة:

> في تحديث دلوقتي في الموقع، بنضيف لكم مميزات جديدة وبنطوّر الخدمات. جرّب تاني بعد لحظات.

تُستخدم صياغات مشتقة وهادئة للحالات التي تحتاج توجيهًا محددًا:

- 404: «الصفحة دي مش متاحة دلوقتي، لكن باقي الموقع شغال بشكل طبيعي.»
- Unauthorized: «سجّل دخولك عشان تكمّل من نفس المكان.»
- Forbidden: «الصفحة دي متاحة لصلاحيات معينة. تقدر ترجع للرئيسية وتكمّل بشكل طبيعي.»
- Session Expired: «جلسة الدخول انتهت للحفاظ على أمان حسابك. سجّل دخولك وكمل.»
- Site Expired: تظل رسالة الاشتراك واضحة وظيفيًا بدون وصفها كعطل تقني.

لا تظهر الكلمات أو التفاصيل التالية للعميل: Stack Trace، Error Code، Internal Server Error، Digest، Request ID، Correlation ID، اسم Exception، أو مسار ملف برمجي.

### 9.2 المكوّن الموحد

`ErrorExperience` يستقبل Variant وسياق الصفحة ويعرض:

- هوية FrameID الهادئة ورسمة/أيقونة غير تحذيرية.
- عنوان ورسالة مطمئنة.
- زر «إعادة المحاولة» ينفذ `window.location.reload()` لضمان Refresh حقيقي.
- زر «الصفحة الرئيسية» يختار الرئيسية المناسبة للمساحة الحالية.
- زر «إبلاغ الإدارة بالمشكلة» ينشئ البلاغ فورًا.
- بعد النجاح، يتحول زر الإبلاغ إلى حالة نجاح ويظهر رقم البلاغ فقط.
- رابط صغير اختياري «إضافة ملاحظة» يفتح حقلًا واحدًا؛ لا يعيق إنشاء البلاغ ولا يكون مطلوبًا.
- حالات Loading وSuccess وAlready Reported وOffline واضحة وهادئة.

يُستخدم المكوّن في الجذر، التسويق، لوحة العميل، لوحة الأدمن، 404، Unauthorized، Forbidden، Session Expired، وأي صفحة خطأ أخرى. نسخة Global Error تحافظ على نفس التصميم بصياغة CSS مستقلة.

## 10. مركز «مشاكل العملاء» داخل الأدمن

### 10.1 الظهور في الأدمن

- عنصر رئيسي باسم «مشاكل العملاء» في Navigation.
- Badge واضح بعدد البلاغات الجديدة.
- بطاقة بارزة في الصفحة الرئيسية تعرض: جديد، قيد المراجعة، محلول، مغلق.
- إشعار داخل Topbar عند وصول بلاغ جديد.
- Error Center الحالي يتحول إلى واجهة هذا المركز مع تبويب/قسم لسجلات الأخطاء التقنية غير المرتبطة ببلاغ.

### 10.2 قائمة البلاغات

تدعم القائمة:

- البحث برقم البلاغ، العميل، البريد، الموقع، Route، Fingerprint، Request ID، أو Correlation ID.
- الفلترة بالحالة، الأولوية، المصدر، المسؤول، العميل، الموقع، والبيئة.
- ترتيب الجديد والأولوية والتحديث الأخير.
- إظهار عدد مرات التكرار وآخر ظهور بدون كشف Stack داخل الجدول.

### 10.3 صفحة البلاغ

تعرض:

- رقم البلاغ، الحالة، الأولوية، تاريخ الإنشاء وآخر تحديث.
- العميل وبيانات التواصل والموقع والـTemplate.
- نوع الخطأ وRoute وURL والبيئة والمتصفح والجهاز.
- Request ID وCorrelation ID وDigest وStack وMetadata داخل أقسام تقنية محمية.
- الكود المتأثر المستخرج من أول Stack Frame صالح.
- المسؤول وسجل الحركة ومرات الظهور المرتبطة.

الأفعال المتاحة:

- بدء المراجعة وتعيين المنفذ الحالي.
- تعليم كمحلول مع Resolution Note داخلية اختيارية.
- إعادة فتح البلاغ.
- إغلاق البلاغ بعد الحل.
- نسخ جميع التفاصيل بعد تنقيتها.
- فتح العميل، الموقع، وسجل الأخطاء.
- فتح الملف البرمجي فقط عند توفر Repository URL وCommit SHA ومسار داخل المشروع؛ يُنشأ رابط إلى نسخة الكود المطابقة. إذا لم تتوفر هذه البيانات يظهر الفعل معطلًا بدل إنشاء رابط غير صحيح.
- إرسال إشعار الحل للعميل إذا كان البلاغ مرتبطًا بـTenant.

## 11. دورة الحالة

الانتقالات المسموحة:

- `NEW -> IN_REVIEW`
- `IN_REVIEW -> RESOLVED`
- `RESOLVED -> CLOSED`
- `RESOLVED -> IN_REVIEW` لإعادة الفتح
- `CLOSED -> IN_REVIEW` لإعادة الفتح

بدء المراجعة يعيّن الأدمن الحالي تلقائيًا ما لم يُحدد مسؤول آخر. الحل يسجل الوقت والمنفذ. إرسال إشعار العميل فعل مستقل بعد الحل حتى لا يرسل النظام رسالة غير مقصودة.

## 12. الأداء والاعتمادية

- لا يجب أن يفشل مسار العميل بسبب فشل إرسال البلاغ؛ الالتقاط Best Effort مع Retry محدود و`keepalive`.
- عمليات إنشاء البلاغ والربط بالسجل وEvent تتم داخل Transaction.
- تُضاف Indexes على `status`, `priority`, `fingerprint`, `tenantId`, `siteId`, `assigneeAdminUserId`, و`createdAt`.
- تُقيد الاستعلامات بصفحات ولا تُحمّل Stack وMetadata الثقيلة في القائمة.
- فشل Logger في قاعدة البيانات يستمر في Console المنظم ولا يدخل في حلقة تسجيل أخطاء ذاتية.

## 13. الاختبارات المطلوبة

### وحدات

- Fingerprint ثابت مع الرسائل المتغيرة، ومختلف عند اختلاف المصدر الحقيقي.
- Sanitizer يحذف الأسرار وQuery Parameters الحساسة.
- مصنف المتصفح والجهاز ونظام التشغيل والاتصال.
- آلة انتقال حالة البلاغ ترفض الانتقالات غير المسموحة.
- سياسة الأولوية والتصعيد الداخلي.

### تكامل

- إنشاء Error Occurrence وربطه بالسياق الموثوق.
- إنشاء بلاغ من ظهور موجود.
- دمج البلاغ المتكرر لنفس العميل/الموقع خلال النافذة.
- عدم دمج بلاغات عملاء أو مواقع مختلفة.
- ربط جلسة العميل والموقع المنشور والأدمن الداخلي بشكل صحيح.
- إشعار الأدمن عند البلاغ الجديد وإشعار العميل عند الحل.
- حفظ اسم الأدمن وأوقات بدء المراجعة والحل والإغلاق وإعادة الفتح.

### واجهة

- كل صفحات الأخطاء تستخدم تجربة موحدة ولا تعرض تفاصيل تقنية.
- Refresh والرئيسية والإبلاغ تعمل في كل Variant.
- الإبلاغ لا يتطلب ملاحظة.
- حالات النجاح، التكرار، Offline، وفشل الإرسال واضحة.
- Badge وإحصائيات الصفحة الرئيسية تتطابق مع الحالات الفعلية.
- صفحة التفاصيل تنفذ كل الأفعال حسب الصلاحية.

### تحقق نهائي

- `npm run typecheck`
- `npm run lint`
- `npm test`
- `npm run build`
- فحص يدوي لمسارات التسويق، لوحة العميل، الأدمن، موقع منشور، API، وServer Action.

## 14. خطة الترحيل والتوافق

1. إضافة الجداول والحقول الجديدة مع Migration غير مدمرة للبيانات الحالية.
2. إبقاء سجلات `ErrorLog` القديمة قابلة للعرض حتى إذا كانت حقولها الجديدة فارغة.
3. تحويل Error Center الحالي إلى Customer Issue Center مع قسم للسجل التقني.
4. توصيل `processError()` وClient Diagnostics بالخدمة الجديدة.
5. استبدال صفحات الأخطاء بالمكوّن الموحد.
6. تحويل Wrappers وEndpoints الحالية تدريجيًا، ثم إجراء بحث شامل عن `console.error`, رسائل الأخطاء التقنية، وملفات `error.tsx` غير الموحدة.
7. لا تُحذف `SupportCase` أو بيانات Error Center القديمة.

## 15. معايير القبول

- جميع صفحات الأخطاء موحدة بصريًا ولغويًا.
- لا تظهر تفاصيل تقنية أو عبارات مقلقة للمستخدم النهائي.
- يمكن إنشاء البلاغ بضغطة واحدة دون كتابة.
- تُجمع بيانات الطلب والجهاز والحساب والموقع والإصدار تلقائيًا عند توفرها.
- كل الأخطاء تمر بمسار الالتقاط الموحد، وكل البلاغات تمر بمركز واحد.

- يمكن تتبع البلاغ عبر الحالات الأربع مع المسؤول وسجل الحركة.
- يظهر البلاغ الجديد في Navigation والصفحة الرئيسية ومركز مشاكل العملاء وإشعارات الأدمن.

- تُدمج البلاغات المتكررة دون فقد مرات الظهور التقنية.
- تبقى البيانات الحساسة محذوفة، والتفاصيل التقنية محصورة بصلاحيات الأدمن.
