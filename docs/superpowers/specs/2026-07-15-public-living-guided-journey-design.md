# كيف يقود الموقع العام المستخدم إلى إنشاء موقعه

| الحقل | القيمة |
|---|---|
| التاريخ | 2026-07-15 |
| الحالة | نسخة نهائية للمراجعة |
| نوع الوثيقة | تصميم تجربة ومواصفات سلوك |
| `meta.contentType` | Conceptual |
| النطاق | الموقع العام فقط |
| خارج النطاق | Smart Tips داخل Dashboard وأي إعادة تصميم للوحة التحكم |

تحدد هذه الوثيقة رحلة توجيه حية داخل موقع FrameID العام. تبدأ الحركة من العنصر المطلوب، ثم تنشئ الرسالة بجواره، ثم تعيد الانتباه إلى العنصر بعد إغلاق الرسالة. لا تستخدم الرحلة Popup أو Modal أو Coach Marks أو طبقة تعتيم.

## نتيجة التصميم

يقود الموقع الزائر عبر المسار التالي:

1. يضغط **ابدأ مجانًا** في الصفحة الرئيسية
2. يختار قالبًا من شبكة القوالب
3. يعاين القالب كأنه موقع حقيقي
4. يضغط **استخدام هذا القالب**
5. يكتب بياناته ويضغط **إنشاء موقعي**
6. ينشئ FrameID الموقع بالقالب المختار وينقله إلى Dashboard

تظل كل صفحة قابلة للاستخدام أثناء التوجيه. الرسالة لا تحجب المحتوى، ولا تستولي على التركيز، ولا تمنع المستخدم من الضغط على العنصر المستهدف.

## الجمهور والهدف

تخدم الرحلة مصورًا يريد موقعًا لكنه لا يعرف ترتيب الخطوات. قد يدخل من الصفحة الرئيسية، أو يفتح صفحة القوالب مباشرة، أو يصل إلى رابط معاينة.

يجب أن يحقق التصميم هدفين:

- يوضح الفعل التالي خلال ثوانٍ من دون شرح الصفحة
- يحافظ على إحساس Premium يليق بقوالب المصورين ولا ينافس الصور على الانتباه

## نطاق الرحلة العامة

تشمل النسخة الأولى الصفحات واللحظات التي تقود مباشرة إلى إنشاء الموقع:

- الصفحة الرئيسية
- صفحة القوالب
- بداية معاينة القالب
- لحظة استخدام القالب بعد استكشاف المعاينة
- صفحة التسجيل وإنشاء الموقع

لا تضيف النسخة الأولى رسائل إلى تسجيل الدخول أو استعادة كلمة المرور أو الصفحات القانونية. هذه الصفحات لا تنتمي إلى مسار إنشاء موقع جديد، وتحتاج وثيقة منفصلة وموافقة جديدة.

## ترتيب الأفعال داخل المسار

تقوي الواجهة المسار الأساسي من دون إجبار المستخدم على Wizard:

- يظل **ابدأ مجانًا** في الرئيسية هو المدخل الأساسي إلى القوالب
- يصبح فتح المعاينة هو الفعل الأساسي داخل كل بطاقة قالب
- يبقى الانتقال المباشر إلى التسجيل فعلًا ثانويًا للمستخدم الذي حسم اختياره
- لا توقظ حركة الـCascade زر التسجيل المباشر، بل توقظ القالب ككل
- يقود زر **استخدام هذا القالب** داخل المعاينة إلى التسجيل مع Template Code نفسه
- تعرض صفحة التسجيل القالب المختار حتى لا ينقطع سياق القرار

## قراءة التجربة الحالية

### الرحلة الحالية

تبدأ الصفحة الرئيسية بزر أساسي وزر ثانوي، وكلاهما يفتح `/templates`. تعرض صفحة القوالب زرين لكل قالب: المعاينة، أو الانتقال المباشر إلى التسجيل. تعرض المعاينة موقع القالب كاملًا مع شريط ثابت يحتوي على **رجوع** و**استخدام هذا القالب**. يحمل رابط التسجيل كود القالب في `?template=noir-gold`، لكن صفحة التسجيل لا تعرض القالب المختار بصريًا.

ينشئ التسجيل الحساب والموقع والقالب والمحتوى الافتراضي داخل عملية واحدة. ينشر النظام الموقع تلقائيًا، ثم يحول المستخدم إلى `/dashboard?welcome=1`.

### نقاط القوة الحالية

- زر البداية في الصفحة الرئيسية واضح ويرتبط بصفحة القوالب
- المستخدم يستطيع معاينة القالب كمنتج حقيقي قبل التسجيل
- اختيار القالب يستمر تقنيًا حتى إنشاء الموقع
- التسجيل ينشئ موقعًا منشورًا بدل إنشاء حساب فارغ
- القوالب تعمل على الموبايل والديسكتوب

### نقاط الضعف الحالية

- يستطيع المستخدم تخطي المعاينة، لذلك لا توجد رحلة أساسية متماسكة
- لا تعرض صفحة التسجيل القالب الذي اختاره المستخدم، فتختفي استمرارية القرار
- لا يوجد رابط مرئي بين الفعل في صفحة والفعل التالي في الصفحة الجديدة
- شريط **استخدام هذا القالب** ثابت منذ بداية المعاينة، لذلك لا يعرف النظام متى انتهى المستخدم من الاستكشاف
- التحليلات الحالية تسجل Page Views فقط، ولا تقيس لحظات الرحلة أو التحويل بين مراحلها
- الرسائل العامة الحالية تظهر في منتصف الشاشة وتغطي العنصر الذي تريد شرحه
- طبقة التعتيم والـBlur تجعل الرسالة تبدو كـModal
- الضغط على زر الرسالة يسجلها كمشاهدة دائمة حتى من دون Checkbox
- الرسائل الحالية مرتبطة بالمسار، لا بالفعل الذي قام به المستخدم
- اللون البنفسجي في رسالة القوالب ينفصل بصريًا عن هوية FrameID الذهبية
- على الموبايل، تغطي الرسالة جزءًا كبيرًا من الشاشة وتتزاحم مع الـCTA الثابت

## ما نتعلمه من Smart Tips داخل Dashboard

يبقى نظام Smart Tips داخل Dashboard كما هو. لا تعيد هذه الوثيقة تصميمه ولا تقترح تعديل محتواه أو حركته أو تخزينه.

يحمل النظام الحالي فلسفة مفيدة:

- لكل صفحة رسالة واحدة ذات هدف واضح
- النص يشرح أهمية الفعل، لا أسماء عناصر الواجهة فقط
- الرسالة ترتبط بعنصر حقيقي داخل الصفحة
- اللهجة المصرية تقلل المسافة بين المنتج والمستخدم
- لكل صفحة Accent وحركة تناسب سياقها
- الذاكرة تمنع التكرار عندما يختار المستخدم ذلك

لا تعيد الرحلة العامة استخدام مكونات Smart Tips أو سجلها أو مفاتيح تخزينها. تأخذ المبدأ فقط: رسالة سياقية قصيرة تقود إلى فعل.

## المبدأ المركزي: العنصر يتكلم

لا تظهر الرسالة أولًا. يبدأ العنصر المستهدف بالحركة، ثم ينشئ المسار، ثم تظهر الرسالة في نهايته.

يمر كل توجيه بمرحلتين:

1. **ولادة الرسالة**: ينتبه العنصر، يظهر حوله أثر ضوئي، يتحرك الضوء إلى مساحة قريبة، ثم تتكون الرسالة
2. **تسليم الرسالة**: تختفي الرسالة، يعود Glint واحد عبر المسار إلى العنصر، يبقى الـHalo حوله نحو ثانية، ثم يهدأ

تجعل المرحلتان العنصر هو المصدر والنهاية. لا يشعر المستخدم أن طبقة خارجية توقفت لتشرح له الموقع.

## مصطلحات النظام

تستخدم الوثيقة المصطلحات التالية بمعنى ثابت:

| المصطلح | المعنى |
|---|---|
| المصدر | العنصر الذي يجب أن ينتبه إليه المستخدم ويضغطه |
| لحظة التوجيه، Journey Moment | التسلسل الكامل من استيقاظ المصدر إلى اختفاء الأثر |
| Halo | أثر ضوئي محدود يحيط بالمصدر |
| Glint | نقطة ضوء واحدة تتحرك على المسار لتوضح الاتجاه |
| الرسالة | السطح الصغير الذي يحمل النص والزر والـCheckbox |
| التسليم، Handoff | حركة العودة من الرسالة إلى المصدر بعد الإغلاق |

## قواعد التجربة

تطبق كل لحظة توجيه القواعد التالية:

- يبدأ العنصر المستهدف قبل الرسالة
- تظهر لحظة واحدة فقط في الوقت نفسه
- لا يوجد تعتيم عام أو Blur على الصفحة
- لا توجد أسهم مستقيمة أو Arrowheads أو دوائر Spotlight
- لا يوجد رقم خطوة أو Progress Bar أو زر **التالي**
- يغلق زر الرسالة التوجيه الحالي فقط، بينما يبقى الفعل التالي داخل المصدر نفسه
- لا تمنع الرسالة الضغط على العنصر المستهدف
- لا تتحرك الصفحة أو يتغير Layout عند ظهور الرسالة
- لا تستخدم الحركة حلقة مستمرة
- تتوقف الحركة إذا بدأ المستخدم الفعل قبل اكتمالها
- يظهر الكارت قريبًا من المصدر، لا في منتصف الشاشة
- يظل طول المسار قصيرًا حتى تبقى العلاقة بين المصدر والرسالة واضحة

## دورة حياة لحظة التوجيه

تدير كل لحظة State Machine مستقلة:

| الحالة | السلوك |
|---|---|
| `idle` | لا يوجد أثر مرئي |
| `waiting-for-source` | ينتظر النظام ظهور العنصر واستقرار مكانه |
| `source-awake` | ينفذ العنصر Breath واحدة أو Cascade قصيرة |
| `halo-visible` | يظهر Halo محدود حول المصدر |
| `message-birth` | يخرج المسار من المصدر وتتكون الرسالة عند نهايته |
| `message-active` | يقرأ المستخدم الرسالة ويستطيع استخدام الصفحة |
| `message-leaving` | تختفي الرسالة أولًا |
| `handoff` | يعود Glint إلى المصدر ويبقى الـHalo نحو ثانية |
| `complete` | يختفي الأثر ولا يعاد داخل العرض الحالي للصفحة |

إذا ضغط المستخدم العنصر أثناء أي حالة، يلغي النظام بقية الحركة فورًا ويسمح بالفعل الأصلي بالاستمرار.

## الإيقاع الحركي المشترك

تحافظ كل الصفحات على إيقاع واحد حتى تبدو الرحلة نظامًا واحدًا:

| اللحظة | المدة المستهدفة |
|---|---:|
| انتظار الصفحة بعد استقرار المصدر | 400 إلى 600 ms |
| Breath أو استيقاظ المصدر | 480 إلى 650 ms |
| ظهور الـHalo | 220 إلى 320 ms |
| انتقال الضوء من المصدر | 320 إلى 480 ms |
| تكوين الرسالة | 260 إلى 380 ms |
| اختفاء الرسالة بعد الضغط | 180 إلى 240 ms |
| عودة الـGlint إلى المصدر | 240 إلى 360 ms |
| بقاء الـHalo بعد الرسالة | 900 إلى 1100 ms |
| اختفاء الـHalo | 350 إلى 500 ms |

يستخدم النظام منحنيات Spring هادئة أو `cubic-bezier(0.16, 1, 0.3, 1)`. لا يستخدم Bounce واضحًا، ولا Overshoot كبيرًا، ولا وميضًا سريعًا.

## تعريف جاهزية الصفحة

لا تعتمد الرحلة على Timeout ثابت وحده. تبدأ بعد تحقق الشروط التالية:

1. أكملت الصفحة أول Paint مفيد
2. ظهر العنصر المستهدف داخل DOM
3. استقر Bounding Box للعنصر خلال Frameين متتاليين
4. أصبح العنصر داخل Viewport بنسبة مناسبة
5. انتهت الصور الحرجة من حجز مساحتها، حتى لو استمر تحميل الصورة نفسها

بعد تحقق الشروط، ينتظر النظام من 400 إلى 600 ms. يمنع ذلك ظهور الحركة أثناء تغير الخطوط أو الصور أو انتقال الصفحة.

## الهوية الواحدة والشخصيات المختلفة

تستخدم كل الرسائل نفس الهيكل:

- خط Tajawal المستخدم في الموقع
- سطح داكن شفاف أو Surface فاتح حسب خلفية الصفحة
- Border داخلي رفيع يوضح حافة المادة
- Radius بين 14 و16 px
- Shadow ملون بلون الخلفية، لا ظل أسود منفصل
- عرض بين 280 و340 px على الديسكتوب
- عرض لا يتجاوز `calc(100vw - 32px)` على الموبايل
- نص من سطرين أو ثلاثة
- زر واحد فقط
- Checkbox صغير بعنوان **ماتظهرش الرسالة دي تاني**
- نقطة اتصال ضوئية صغيرة تربط الرسالة بمصدرها من دون أي Arrowhead

تتغير الشخصية عبر طريقة ولادة الرسالة، لا عبر بناء كارت جديد من الصفر.

### شخصية البداية في الصفحة الرئيسية

الاسم الداخلي: `Ignition`

يعبر زر **ابدأ مجانًا** عن لحظة البداية. ينفذ الزر Breath واحدة بنسبة Scale لا تتجاوز `1.018`. يظهر Halo بلون Champagne حول حدود الزر، ثم يخرج مسار منحني قصير من إحدى زواياه إلى مساحة فارغة بجواره.

تتكون الرسالة من نقطة الاتصال إلى الخارج. يظهر أولًا خط الحافة، ثم سطح الرسالة، ثم النص والزر. لا تتحرك الرسالة أكثر من 8 px.

### شخصية الإبداع في صفحة القوالب

الاسم الداخلي: `Prism Cascade`

تستيقظ شبكة القوالب بدل تحديد بطاقة واحدة. تنفذ أول ثلاث بطاقات ظاهرة Cascade خفيفة بالتتابع. تستخدم كل بطاقة `opacity` و`transform` فقط، مع انتقال رأسي من 6 إلى 8 px وScale يبدأ من `0.992`.

بعد انتهاء البطاقة الأخيرة، تتجمع Reflection رفيعة عند الحافة المشتركة للشبكة، ثم تنشئ الرسالة بجوار الشبكة. لا يوجد سهم، ولا Border حول الشبكة، ولا Spotlight على بطاقة بعينها.

إذا ظهرت بطاقتان فقط، ينفذ النظام مرحلتين. لا يصنع عنصرًا وهميًا لإكمال الإيقاع.

### شخصية الواقعية في المعاينة

الاسم الداخلي: `Screen Voice`

تتعامل الرحلة مع المعاينة كشاشة حقيقية تتحدث. يمر Reflection خافت على حافة شاشة المعاينة أو إطار الهاتف. يخرج المسار من الحافة، ثم تتكون الرسالة خارج مساحة المحتوى المهمة.

على الديسكتوب، تظهر الرسالة قرب ركن الشاشة الأقل ازدحامًا. على الموبايل، تظهر الرسالة فوق شريط المعاينة الثابت أو بجوار حافة مرئية من القالب. لا تظهر في منتصف الشاشة ولا تغطي وجهًا أو صورة رئيسية متى أمكن اكتشاف ذلك من موضع المحتوى.

### شخصية الانتقال عند استخدام القالب

الاسم الداخلي: `Threshold`

لا تبدأ هذه اللحظة عند فتح المعاينة. تنتظر حتى يستكشف المستخدم القالب. يتحقق ذلك عند وصوله إلى الجزء الأخير من المحتوى، أو بعد Scroll فعلي يتجاوز 55% من طول الصفحة.

ينفذ زر **استخدام هذا القالب** Directional Sheen واحدة، ثم يظهر Halo محدود على حافته. يخرج مسار منحني بطول قصير إلى الرسالة. بعد الضغط على **تمام**، يعود Glint عبر المسار باتجاه الزر، ويبقى الـHalo حول الزر نحو ثانية.

توضح حركة العودة أن الفعل التالي موجود داخل الزر، لا داخل الرسالة.

### شخصية التجميع في صفحة التسجيل

الاسم الداخلي: `Assembly`

تعرض صفحة التسجيل القالب المختار بصريًا قرب عنوان النموذج. يكفي Thumbnail صغير واسم القالب وعبارة **القالب اللي اخترته**. لا يتحول هذا التأكيد إلى كارت جديد.

ينشئ زر **إنشاء موقعي** الرسالة بعد استقرار النموذج. عند إرسال النموذج، تتحول حالة الزر إلى تقدم هادئ يوضح أن FrameID ينشئ الحساب والموقع والرابط. لا يظهر احتفال داخل Dashboard، ولا يتغير أي سلوك هناك.

إذا فتح المستخدم `/signup` من دون قالب، تستخدم الصفحة رسالة عامة ولا تعرض تأكيد قالب.

## نصوص الرحلة

تستخدم الرسائل المصرية التالية كنقطة اعتماد أولى. يمكن تحسين كلمة واحدة أثناء التنفيذ إذا أثبت الاختبار أنها أوضح، لكن لا يتغير المعنى أو طول الرسالة من دون مراجعة.

### الصفحة الرئيسية

**النص**

> ابدأ من هنا.
> اختار شكل موقعك، وبعدها هنجهزه باسمك.

**الزر**: تمام

### صفحة القوالب

**النص**

> اختار الشكل الأقرب لشغلك.
> الصور والألوان والكلام كله هتغيره بعدين.

**الزر**: فهمت

### بداية المعاينة

**النص**

> اتفرج عليه بعين عميلك.
> ده الشكل الحقيقي، وكل تفصيلة هتقدر تغيرها بعدين.

**الزر**: فهمت

### زر استخدام القالب

**النص**

> القالب جاهز يبقى موقعك.
> اضغط «استخدام هذا القالب» وابدأ تحط صورك وبياناتك.

**الزر**: تمام

### صفحة التسجيل

**النص**

> فاضل بياناتك.
> اضغط «إنشاء موقعي» وهنجهز القالب والرابط تلقائيًا.

**الزر**: تمام

## منطق الذاكرة والـCheckbox

يخفي الـCheckbox اللحظة الحالية فقط. لا يخفي بقية الرحلة.

يعمل السلوك كالتالي:

- الضغط على **تمام** أو **فهمت** من دون تحديد الـCheckbox يغلق الرسالة داخل العرض الحالي فقط
- إذا غادر المستخدم الصفحة وعاد إليها، تظهر الرسالة مرة أخرى
- تحديد **ماتظهرش الرسالة دي تاني** ثم إغلاق الرسالة يحفظ Moment ID داخل `localStorage`
- لا يستخدم النظام مفتاح Smart Tips الحالي
- لا يزامن الاختيار مع الحساب أو Dashboard
- لا يعيد النظام الرسالة المخفية بعد تعديل محدود في النص
- إذا تغير الهدف أو الفعل نفسه، يحصل التصميم الجديد على Moment ID جديد بدل إعادة استخدام القديم

لا تعرض اللحظة أكثر من مرة داخل Render واحد للصفحة، حتى لو خرج العنصر من Viewport ثم عاد.

## ترتيب اللحظات من دون Wizard

لا تعمل الرحلة كـWizard خطي. كل لحظة مستقلة ويمكنها مساعدة المستخدم الذي دخل من رابط مباشر.

تطبق الأولوية التالية عند وجود أكثر من Trigger:

1. رسالة الصفحة الأساسية
2. رسالة مرتبطة بفعل جديد ظهر بعد تفاعل المستخدم
3. رسالة CTA النهائية

لا تبدأ رسالة **استخدام هذا القالب** قبل إغلاق رسالة بداية المعاينة. إذا لم يغلق المستخدم الرسالة الأولى، لا تتراكم الرسائل ولا تظهر رسالة ثانية خلفها.

## التموضع والمسار

يحسب النظام موضع الرسالة بعد معرفة مساحة المصدر، وحواف الشاشة، والعناصر الثابتة.

يطبق التموضع القواعد التالية:

- مسافة آمنة 16 px من حواف الشاشة على الموبايل
- مسافة آمنة 24 px على الديسكتوب
- مسافة 12 إلى 18 px بين المصدر وبداية المسار
- طول مسار بين 40 و120 px على الديسكتوب
- طول مسار بين 28 و64 px على الموبايل
- المسار لا يمر فوق النص الأساسي أو CTA أخرى، وتحدد كل صفحة Safe Zone مفضلة
- الرسالة لا تغطي المصدر ولا تمنع الضغط عليه
- يتغير اتجاه الرسالة تلقائيًا بين أعلى وأسفل ويمين ويسار
- يبقى الانحناء محدودًا، ولا يتحول المسار إلى شكل زخرفي

لا يحتوي المسار على رأس سهم. يظهر الاتجاه عبر Glint واحد يتحرك من المصدر إلى الرسالة عند الولادة، ثم من الرسالة إلى المصدر عند التسليم.

إذا لم توجد مساحة Overlay آمنة على شاشة ضيقة، تظهر الرسالة Inline بجوار العنصر داخل تدفق الصفحة. يحتفظ هذا البديل بنفس نقطة الاتصال وحركة التسليم، ولا يتحول إلى Bottom Sheet.

## الوصول

تظل الرحلة مفهومة وقابلة للاستخدام من لوحة المفاتيح وقارئات الشاشة:

- لا ينتقل Focus تلقائيًا إلى الرسالة
- ترتبط الرسالة بالمصدر عبر `aria-describedby` أثناء نشاطها
- يعلن النص عبر `aria-live="polite"` مرة واحدة
- يصل المستخدم إلى الـCheckbox والزر بترتيب منطقي
- يغلق زر Escape الرسالة الحالية من دون حفظ الإخفاء الدائم
- يحتفظ المصدر بحالة Focus الأصلية ولا يستبدلها الـHalo
- تمر النصوص والأزرار بنسبة تباين WCAG AA على الأقل
- لا تعتمد دلالة الفعل على اللون أو الحركة وحدهما

## الحركة المنخفضة والشفافية المنخفضة

يحترم النظام `prefers-reduced-motion` بالكامل:

- لا ينفذ Breath أو Cascade أو انتقال Glint
- يظهر Halo ثابتًا من دون Scale
- تظهر الرسالة عبر Fade قصير لا يتجاوز 120 ms
- تختفي الرسالة أولًا، ثم يبقى Outline ثابت حول المصدر نحو ثانية
- لا توجد حركة Loop

عند دعم `prefers-reduced-transparency`، يستخدم النظام Surface مصمتًا بدل `backdrop-filter`. يبقى النص واضحًا حتى إذا لم يدعم المتصفح Blur.

## الأداء

ينفذ النظام الحركة عبر `transform` و`opacity` كلما أمكن حتى يعالجها Browser Compositor على GPU. يستخدم المسار SVG ثابتًا صغيرًا أو CSS Motion Path، بينما يتحرك الـGlint عبر `transform` فقط.

تطبق القيود التالية:

- لا تضاف مكتبة Animation جديدة للنسخة الأولى
- لا يستخدم النظام WebGL أو Canvas أو Three.js
- لا يستخدم `window.scroll` لتحديث React State في كل Frame
- يراقب ظهور المصدر عبر `IntersectionObserver`
- يراقب تغير الحجم عبر `ResizeObserver`
- يكتب الإحداثيات في CSS Custom Properties خارج دورة Render المستمرة
- يستخدم `will-change` أثناء الحركة فقط، ثم يزيله
- يوجد مسار واحد ورسالة واحدة في DOM لكل لحظة نشطة
- لا تستخدم Particles دائمة أو Noise على حاوية Scroll
- لا يزيد Runtime الخاص بالرحلة عن 10 KB مضغوطًا كهدف هندسي

تحافظ التجربة على الحدود التالية:

| المقياس | الحد |
|---|---:|
| Largest Contentful Paint (LCP) | أقل من 2.5s |
| Interaction to Next Paint (INP) | أقل من 200 ms |
| Cumulative Layout Shift (CLS) | أقل من 0.1 |
| تغير CLS بسبب الرحلة | 0 |

## الحالات البديلة

يفشل النظام بهدوء ولا يعطل الصفحة:

- إذا لم يوجد المصدر، لا تظهر الرسالة
- إذا خرج المصدر من DOM أثناء الحركة، تختفي الرسالة والمسار فورًا
- إذا بدأ Route Transition، تلغى كل Timers وObservers
- إذا كان `localStorage` غير متاح، تعمل الرسالة من دون حفظ الإخفاء
- إذا كانت أبعاد المصدر صفرًا، ينتظر النظام حتى يصبح مرئيًا
- إذا تغير حجم الشاشة، يعيد النظام الحساب مرة واحدة بعد استقرار Resize
- إذا ضغط المستخدم CTA قبل ظهور الرسالة، يستمر الانتقال من دون تأخير

## القياس

تضيف الرحلة Events خالية من البيانات الشخصية إلى Google Analytics:

| الحدث | وقت الإرسال |
|---|---|
| `public_journey_moment_seen` | بعد ظهور الرسالة بالكامل |
| `public_journey_message_dismissed` | عند الضغط على زر الرسالة |
| `public_journey_moment_suppressed` | عند الإغلاق مع تحديد الـCheckbox |
| `public_journey_target_activated` | عند الضغط على العنصر المستهدف |
| `template_preview_opened` | عند فتح المعاينة من شبكة القوالب |
| `template_use_clicked` | عند الضغط على استخدام القالب |
| `signup_submitted` | عند إرسال نموذج صالح |
| `site_created` | بعد نجاح إنشاء الموقع |

تشمل Properties المسموح بها Moment ID وRoute وTemplate Code وحالة Reduced Motion. لا تشمل الاسم أو الهاتف أو البريد الإلكتروني أو Slug.

تقيس المراجعة أثر النظام عبر:

- نسبة الضغط على **ابدأ مجانًا** بعد ظهور لحظة البداية
- نسبة فتح المعاينة من صفحة القوالب
- نسبة الضغط على **استخدام هذا القالب** بعد استكشاف المعاينة
- نسبة إكمال التسجيل بعد اختيار قالب
- نسبة إخفاء كل رسالة نهائيًا
- معدل فشل العثور على المصدر
- Core Web Vitals قبل الرحلة وبعدها

## حدود العزل عن Dashboard

Smart Tips داخل Dashboard منطقة مجمدة. لا يغير التنفيذ المستقبلي الملفات أو السلوكيات التالية:

- تعريفات رسائل Dashboard ونصوصها
- مكون `SmartTip` وتصميمه وحركاته
- مكون `SmartTipRouter` ومنطق اختيار الرسالة
- `data-smart-tip` داخل صفحات Dashboard
- مفتاح التخزين الحالي وإعادة الضبط من Settings
- ترتيب الرسائل وألوانها ومواضعها

يحتوي النظام الحالي على رسائل عامة ورسائل Dashboard داخل Router عالمي واحد. يحتاج منع الازدواج في الموقع العام إلى عزل نقطة التركيب فقط:

- يزال التركيب العالمي من Root Layout من دون تعديل ملفات Smart Tips نفسها
- يركب `SmartTipRouter` الحالي كما هو داخل Layout خاص بمجموعة مسارات Dashboard
- يبقى Smart Tips ظاهرًا على مسارات `/dashboard` بالسلوك الحالي نفسه
- يركب `PublicLivingJourney` داخل مجموعة مسارات الموقع العام فقط
- لا يشترك النظامان في Registry أو Component أو CSS أو Storage
- لا يظهر النظامان معًا على أي Route

هذا العزل تغيير احتواء، لا إعادة تصميم لنظام Dashboard. يجب أن يثبت اختبار Regression أن مخرجات Dashboard قبل العزل وبعده متطابقة.

## مكونات النظام المقترحة

تقسم البنية إلى وحدات مستقلة:

| الوحدة | المسؤولية |
|---|---|
| `PublicLivingJourney` | يختار Moment المناسبة للمسار والفعل |
| `JourneySource` | يعلن العنصر الذي يبدأ الحركة ويستقبل الأثر النهائي |
| `JourneyEmitter` | يدير Breath وHalo وGlint والمسار |

| `JourneyTelemetry` | يرسل Events من دون بيانات شخصية |

لا تعرف الصفحة تفاصيل الرسم أو التخزين. تحدد فقط Moment ID والمصدر والنص والشخصية والـTrigger.

## استراتيجية الاختبار

### اختبارات الوحدة

- انتقال State Machine بالترتيب الصحيح
- إلغاء الحركة عند الضغط على المصدر
- حفظ Moment الحالية فقط عند تحديد الـCheckbox
- عدم الحفظ عند الضغط على الزر من دون Checkbox
- أولوية الرسائل عند تعدد Triggers
- عدم تكرار الرسالة داخل عرض الصفحة نفسه

### اختبارات المكونات

- ظهور الرسالة قرب المصدر من الجهات الأربع
- بقاء الـHalo بعد اختفاء الرسالة
- تحول الموبايل إلى Inline عند غياب مساحة آمنة
- مسار Reduced Motion
- Surface مصمت عند Reduced Transparency
- Focus وEscape و`aria-describedby`

### اختبارات الرحلة

- الرئيسية إلى القوالب
- القوالب إلى المعاينة
- المعاينة إلى استخدام القالب
- استخدام القالب إلى التسجيل مع استمرار Template Code
- التسجيل إلى إنشاء موقع منشور
- دخول مباشر إلى القوالب أو المعاينة أو التسجيل
- العودة إلى صفحة مع رسالة غير مخفية
- العودة إلى صفحة مع رسالة مخفية نهائيًا

### اختبارات عدم المساس بـDashboard

- Snapshot لتعريفات Smart Tips الحالية
- اختبار ظهور كل رسالة Dashboard في Route الحالية
- اختبار مفتاح التخزين وإعادة الضبط الحاليين
- فحص Git Diff يمنع تعديل ملفات Smart Tips المجمدة من دون موافقة صريحة

### اختبارات الأداء

- Lighthouse على الرئيسية والقوالب والمعاينة والتسجيل
- Chrome Performance Trace أثناء كل شخصية حركية
- التحقق من غياب Long Tasks فوق 50 ms بسبب الرحلة
- التحقق من عدم حدوث Layout Shift عند ظهور الرسالة أو اختفائها
- اختبار جهاز موبايل متوسط، لا محاكاة Desktop فقط

## معايير القبول

يقبل التصميم للتنفيذ عندما يحقق كل بند:

- يبدأ المصدر الحركة قبل الرسالة في كل Moment
- لا تظهر أي رسالة في منتصف الشاشة كحالة افتراضية
- لا يوجد Backdrop عام أو تعتيم للصفحة
- لا يوجد Arrowhead أو Spotlight دائري
- تختفي الرسالة قبل اختفاء الـHalo
- يبقى الـHalo بين 900 و1100 ms بعد الرسالة
- يستخدم كل كارت زرًا واحدًا وCheckbox واحدًا
- يخفي الـCheckbox الكارت الحالي فقط
- تعود الرسالة عند زيارة جديدة إذا لم يحدد المستخدم الـCheckbox
- تختلف شخصية الصفحات عبر الحركة، لا عبر هوية بصرية منفصلة
- لا تتحرك خصائص Layout أثناء الأنيميشن
- يعمل Reduced Motion من دون فقد المعنى
- لا يتغير Smart Tips داخل Dashboard بصريًا أو سلوكيًا
- لا يبدأ التنفيذ قبل موافقة صاحب المشروع على هذه الوثيقة

## قرارات مستبعدة

استبعد التصميم الخيارات التالية لأنها لا تخدم الفعل التالي:

- Popup أو Modal أو Toast
- Coach Marks وSpotlight Cutouts
- أسهم مباشرة أو دوائر حول العناصر
- جولة خطية بأزرار **التالي** و**السابق**
- Particles مستمرة أو خلفيات ثلاثية الأبعاد
- Magnetic Cursor أو تتبع الماوس داخل الرسالة
- تغيير Accent جذري في كل صفحة
- تشغيل الحركة في Loop لجذب الانتباه بالقوة
- ربط إخفاء الرسائل العامة بإعدادات Dashboard

## بوابة التنفيذ

هذه الوثيقة لا تفوض كتابة الكود. تبدأ خطة التنفيذ فقط بعد مراجعة النسخة النهائية والموافقة الصريحة عليها. تظل ملفات Smart Tips داخل Dashboard مجمدة طوال التنفيذ.
